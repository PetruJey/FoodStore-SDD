## Context

Food Store es un sistema de e-commerce para productos alimenticios desde cero. Actualmente el repositorio está vacío: `backend/` y `frontend/` solo contienen `.gitkeep` y `.env.example`. No existe ninguna estructura de código, configuración de base de datos, o frontend funcional.

Este change establece el scaffolding completo del monorepo, habilitando que las 19 épicas restantes (Sprints 1-8) puedan construirse sobre una base sólida y consistente.

**Stack**: Python FastAPI (backend), React + TypeScript + Vite (frontend), PostgreSQL (BD), SQLModel + Alembic (ORM/migraciones).

## Goals / Non-Goals

**Goals:**
- Estructura de monorepo con `/backend` (feature-first) y `/frontend` (Feature-Sliced Design)
- Backend FastAPI funcional con `main.py`, configuración, CORS, rate limiting, y registro de routers
- Todos los modelos SQLModel del ERD (16 tablas) con migraciones Alembic autogenerated
- Script de seed idempotente con datos iniciales (roles, estados de pedido, formas de pago, admin)
- Frontend React + Vite + TypeScript con Tailwind CSS, TanStack Query, y routing base
- Stores Zustand (auth, cart, payment, ui) con persistencia selectiva en localStorage
- Patrones de infraestructura: `BaseRepository[T]`, `UnitOfWork`, dependencias FastAPI (`get_current_user`, `require_role`)
- Cliente HTTP Axios con interceptores JWT y refresh automático
- Manejo de errores RFC 7807 con exception handlers globales
- `docker-compose.yml` para PostgreSQL, `.env.example` para backend y frontend, `.gitignore`

**Non-Goals:**
- Implementación de features de negocio (autenticación real, CRUDs, carrito funcional, pagos, etc.) — son épicas de Sprints 1-8
- Pruebas unitarias o de integración (se agregan en cada épica correspondiente)
- Despliegue a producción o configuraciones de CI/CD
- Diseño UI/UX más allá del layout base y routing
- Optimización de performance avanzada

## Decisions

### 1. Monorepo con carpetas separadas (no workspaces de monorepo tool)
- **Decisión**: `/backend` y `/frontend` como carpetas independientes, cada una con su propio `package.json`/`requirements.txt`
- **Por qué**: Simplicidad máxima. El proyecto es manejable por un equipo pequeño. No justifica la complejidad de Nx, Turborepo, o Pants. Cada lado tiene su propio tooling y dependencias.
- **Alternativa descartada**: Nx workspace — overkill para el alcance actual.
- **Alternativa descartada**: Poetry monorepo — Poetry no maneja bien múltiples proyectos en un repo.

### 2. Backend: Feature-first modules (no capas horizontales)
- **Decisión**: Cada módulo de negocio (auth, productos, pedidos, etc.) tiene sus propios archivos `model.py`, `schemas.py`, `repository.py`, `service.py`, `router.py`
- **Por qué**: Cohesión alta. Un desarrollador que trabaja en "productos" encuentra todo en un solo lugar. Escala bien a medida que crecen los módulos.
- **Alternativa descartada**: Capas horizontales tipo `models/`, `services/`, `routers/` — a medida que crece el proyecto, es imposible navegar sin saber qué archivo contiene qué.

### 3. SQLModel sobre SQLAlchemy puro
- **Decisión**: Usar SQLModel para modelos de BD que también funcionan como schemas Pydantic
- **Por qué**: Creado por el autor de FastAPI, elimina la duplicación modelo-schema, integración natural con FastAPI. Para casos complejos (DTOs específicos), se usan Pydantic models separados.
- **Riesgo**: SQLModel es más joven que SQLAlchemy. Para queries complejas, podemos caer a SQLAlchemy core directamente (soportado).

### 4. Zustand sobre Redux/Context
- **Decisión**: Zustand para estado cliente (auth, carrito, UI) + TanStack Query para estado servidor
- **Por qué**: API minimalista, suscripciones granulares (performance), middleware de persistencia built-in, funciona fuera de React (útil para interceptores Axios). TanStack Query maneja fetching, caching y revalidación — elimina la necesidad de stores para datos del servidor.
- **Alternativa descartada**: Redux Toolkit — demasiado boilerplate para este alcance. React Context — re-renders innecesarios.

### 5. Axios con interceptores sobre fetch nativo
- **Decisión**: Axios instance centralizada con interceptores para JWT y refresh automático
- **Por qué**: Los interceptores permiten manejo transversal de autenticación (adjuntar token, refresh en 401, reintentar request original). fetch nativo requiere wrapper manual para esto.
- **Alternativa descartada**: ky (fetch-based) — más moderno pero menos ecosistema y documentación.

### 6. Alembic autogenerate desde modelos SQLModel
- **Decisión**: Migraciones generadas automáticamente desde los modelos SQLModel, con revisiones manuales para casos complejos
- **Por qué**: Los modelos son la fuente de verdad. Alembic detecta cambios y genera el DDL. Para cambios estructurales complejos (migraciones de datos, renombres), se edita la migración manualmente.

### 7. Seed script como módulo Python ejecutable
- **Decisión**: `python -m scripts.seed` ejecuta la carga de datos iniciales con `INSERT ... ON CONFLICT DO NOTHING`
- **Por qué**: Idempotente, ejecutable desde CLI, sin necesidad de comandos mágicos. Los IDs son explícitos (no autogenerados) para que el código pueda referenciarlos.

### 8. Docker Compose solo para PostgreSQL
- **Decisión**: `docker-compose.yml` levanta únicamente PostgreSQL. Backend y frontend se ejecutan localmente con `uvicorn` y `npm run dev`.
- **Por qué**: Durante desarrollo es más rápido iterar sin rebuild de Docker. En producción se containeriza todo, pero eso es post-MVP.

## Risks / Trade-offs

- **[Dependencia externa] SQLModel aún evoluciona rápido** → Mantener SQLAlchemy como opción de escape. Si SQLModel introduce breaking changes, la migración a SQLAlchemy puro es directa (SQLModel extiende SQLAlchemy).
- **[Complejidad] 16 tablas en el primer migration** → La migración inicial será grande. Si falla, `alembic downgrade -1` revierte todo. Probar contra BD vacía antes de commit.
- **[Scope creep] Tentación de empezar features de negocio en este change** → Mantener foco estricto en infraestructura. Cero lógica de negocio. Este change es la base, no la casa.
- **[Frontend] FSD puede sentirseover-engineered para un proyecto chico** → FSD da estructura que evita el caos a medida que crece. Las 5 capas (app, pages, widgets, features, entities, shared) están justificadas para 77 historias de usuario.
- **[Seed] Datos hardcodeados en código** → Los datos de seed son estables (roles, estados). Si necesitan cambiar, se modifica el script. No justifica una tabla de configuración dinámica para 4 roles y 6 estados.
